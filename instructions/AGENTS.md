# 공통 규칙

## 코드를 고치기 전에

- 고치려는 파일과 그 주변 코드를 먼저 읽는다. 추측으로 수정하지 않는다.
- 새로 만들기 전에 이미 있는 컴포넌트·유틸·훅을 먼저 찾는다.
- 라이브러리를 임의로 추가하지 않는다. 필요하면 먼저 묻는다.
- 규모가 있는 작업은 먼저 계획을 세우고 시작한다.

## 코드를 쓸 때

- 주변 코드의 스타일·네이밍·구조를 따른다. 나만의 패턴을 새로 만들지 않는다.
- 불필요한 추상화를 만들지 않는다. 지금 필요한 것만 만든다.
- 자명한 코드에 주석을 달지 않는다. 왜 그렇게 했는지가 안 보일 때만 쓴다.
- 타입 오류를 남기지 않는다.

## 끝내기 전에

- 관련 테스트와 lint, typecheck 를 실행한다.
- 실패하면 실패했다고 말한다. 통과한 척하지 않는다.
- 실행하지 못한 검증이 있으면 무엇을 못 했는지 명시한다.
- 마지막에 변경한 파일과 검증 결과를 한글로 요약한다.

# Codex 전용 규칙

## AGENTS.md 범위

- 작업을 시작하기 전에 대상 디렉터리에 적용되는 `AGENTS.md` 를 확인한다.
  더 가까운 경로의 `AGENTS.md` 가 상위 것보다 우선한다.
- 프로젝트 `AGENTS.md` 의 규칙이 이 전역 파일과 충돌하면 **프로젝트 쪽을 따른다.**
- 프로젝트에 빌드·테스트 명령이 적혀 있으면 임의로 다른 명령을 만들지 말고 그대로 쓴다.

## 변경 검토

- 작업을 마치기 전에 `git diff` 로 변경 전체를 직접 훑는다.
- 의도하지 않은 파일이 섞였는지, 디버그 출력이나 임시 코드가 남았는지 확인한다.
- 변경 범위가 요청보다 넓어졌으면 왜 넓어졌는지 설명한다.

## 승인과 샌드박스

- `approval_policy` 와 `sandbox_mode` 가 무엇으로 설정돼 있든, 되돌리기 어려운 작업은 먼저 알린다.
  (파일 삭제, force push, 원격 반영, 외부 서비스 호출)
- 샌드박스 때문에 실패한 명령은 조용히 우회하지 말고 무엇이 막혔는지 보고한다.

## 보고

- 테스트하지 못한 항목은 "테스트 안 함" 이라고 명시한다. 추정으로 통과를 말하지 않는다.
- 명령 출력이 근거인 경우 실제 출력을 인용한다.
